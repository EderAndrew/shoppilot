# Implementation Plan: Archived Lists, Tab Bar & Reusable Product Search

**Branch**: `003-archived-lists-tabbar` | **Date**: 2026-05-06 | **Spec**: [spec.md](./spec.md)  
**Input**: Feature specification from `specs/003-archived-lists-tabbar/spec.md`

## Summary

Restructure the mobile app's navigation to use a Tab Bar (Lists / Archived / User), filter archived lists out of the main screen, introduce a dedicated Archived screen with month/year dates, migrate logout to the User tab, and fix the ProductPicker to remove the dropdown list UI while keeping reusable product search behavior.

All changes are additive and low-risk: route restructuring, application-layer filtering, minimal UI adjustments. No database migrations, no new domain entities, no library changes.

---

## Navigation Strategy

### Route organization

Expo Router's `(tabs)` group convention is used. The `(app)` group becomes a Stack that hosts both the tabs group and any presentation (modal) screens above the tabs:

```
(app)/_layout.tsx     → Stack layout (auth guard + modal container)
  └── (tabs)/_layout.tsx  → Tabs navigator (3 tabs)
        ├── lists/_layout.tsx     → Stack
        ├── archived/_layout.tsx  → Stack
        └── user/index.tsx        → Single screen (no stack needed initially)
  └── products/new.tsx            → Modal screen (presented over tabs)
```

### Stack per tab

Each tab has its own Stack navigator via `_layout.tsx`. This ensures:
- Back navigation within a tab doesn't affect other tabs
- Deep linking into `lists/[listId]/item-new` works independently
- Future tabs can be added by creating a new directory under `(tabs)/`

### Adding future tabs

Add a new directory under `(app)/(tabs)/` with its own `_layout.tsx` (Stack) and `index.tsx`. No changes to existing tabs needed. Update `(tabs)/_layout.tsx` to add the new `<Tabs.Screen>` entry.

## State & Cache Strategy

### Active vs Archived query separation

```ts
// queryKeys.ts additions
shoppingLists: {
  all:      () => ["shoppingLists"],
  active:   () => ["shoppingLists", "active"],
  archived: () => ["shoppingLists", "archived"],
}
```

Each tab uses its own query — no shared state pollution between tabs.

### Invalidation after archive

The `useArchiveShoppingListMutation` onSuccess currently invalidates `queryKeys.shoppingLists.all()`. This parent key prefix invalidates both `active` and `archived` sub-queries automatically via TanStack Query's prefix matching. No changes to mutation invalidation needed.

### Avoiding unnecessary re-fetches

- `staleTime` remains at the existing default for list queries
- Each tab's query is independent — switching tabs does not re-fetch if data is fresh
- `ShoppingListCard` is rendered via `FlatList` for virtualization (already in place)

### Phase 6 — ProductPicker Fix

**Objective**: Remove the dropdown list of products from the add-item form. Keep the reusable product search logic but change the UI to show only an inline suggestion or an absence message.

**Scope**:

| Action | File |
|--------|------|
| MODIFY | `apps/mobile/src/features/products/ProductPicker.tsx` — remove list rendering; add three UI states |

**New ProductPicker behavior**:

```
State 1: query is empty (or too short)
  → Render: nothing below the input

State 2: query is non-empty + products.data has results
  → Use first match to pre-fill or suggest via onSelect
  → Render: inline hint text or auto-select behavior (no dropdown list)

State 3: query is non-empty + products.data is empty
  → Render: <Text color="$colorSubtle">Nenhum produto reutilizável encontrado</Text>
```

The search query logic (`useProductsQuery` or equivalent) and `onSelect` callback remain unchanged — only the rendered output changes.

**Dependencies**: None — this is an isolated UI component change.

**Risks**:
- Users who relied on seeing the list to browse/select products lose that ability. However, the spec explicitly requires removing the visual list. The inline suggestion for the first match mitigates discoverability loss.
- If the product picker is used in multiple places beyond the add-item form, verify no other screen breaks.

**Rollback**: Revert `ProductPicker.tsx` to previous version.

**Tests**:
- Manual: Open add-item form → type a product name that exists → confirm no list appears, first match is suggested
- Manual: Type a product name with no match → confirm only "Nenhum produto reutilizável encontrado" text appears
- Manual: Clear the input → confirm no text or list appears below
- Manual: Select a suggested product → confirm form populates correctly

**Validation criteria**:
- No list of products rendered below the input at any time
- Matching a reusable product shows a single inline suggestion
- No match shows only the "Nenhum produto reutilizável encontrado" message
- The full add-item flow (with product selection) still completes correctly

Each phase delivers independently testable value:

| Phase | Deliverable | Standalone value |
|-------|-------------|-----------------|
| 2 | Tab Bar skeleton | App has 3-tab structure; existing screens work in new location |
