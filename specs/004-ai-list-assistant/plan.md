# Implementation Plan: AI Shopping List Assistant

**Branch**: `004-ai-list-assistant` | **Date**: 2026-05-15 | **Spec**: [spec.md](./spec.md)
**Input**: Feature specification from `specs/004-ai-list-assistant/spec.md`

---

## Summary

Introduce an AI-powered shopping suggestion flow accessible from active list detail screens. The user types a natural language prompt ("churrasco para 10 pessoas"), the system returns a structured list of product suggestions via a Supabase Edge Function, and the user selects which items to add before anything is persisted.

No existing flows are modified. The manual `+ Item` flow remains unchanged. The AI assistant is additive only — it is an alternative entry point that feeds into the same `AddShoppingListItem` use case as manual item creation.

No database migrations are required.

**Storage**: Supabase (PostgreSQL via RLS) — no schema changes
**Testing**: Vitest (node env) — `pnpm --filter mobile test`
**Target Platform**: iOS + Android + Web (React Native / Expo)
**Constraints**: API key must never appear in the mobile bundle; AI failure must not block manual item addition; no automatic item insertion

## Architecture Strategy

### Provider independence (BYO-AI path)

The `AIRepository` port is the single seam for AI integration. To swap providers:
1. Replace `SupabaseAIRepository` with any class that implements `AIRepository`
2. Change the Edge Function implementation (or deploy a new function)
3. Zero changes to application, domain, or UI code

For a future "bring your own AI" feature, the user's custom endpoint can be stored in a settings table and passed to a new `CustomAIRepository` implementation — the port and use case remain unchanged.

### Reuse across future features

The `ai-assistant` feature directory, `AIRepository` port, and Edge Function can be reused for:
- "Complete my list with pantry basics"
- "Suggest items based on last month's list"
- Recipe-to-shopping-list conversion

All of these are future `SuggestShoppingListItems` input variants — no new ports or infrastructure adapters needed.

## State & Cache Strategy

### AI request lifecycle state

```
Zustand uiStore.isAIAssistantOpen    → controls sheet mount/unmount
useMutation (useAISuggestMutation):
  status = "idle"                     → initial, no request sent
  status = "pending"                  → waiting for Edge Function
  status = "success"                  → suggestions received, show review UI
  status = "error"                    → show error, allow retry
AISuggestionSheet local state:
  selectedIds: Set<string>            → which suggestion ids are checked
```

### Cache invalidation after confirmation

The confirmation hook (`useConfirmSuggestions`) calls `AddShoppingListItem` N times for N selected suggestions, then invalidates `queryKeys.shoppingLists.detail(listId)` once after all mutations resolve. This single invalidation triggers one refetch of the list detail, updating budget totals and item list in one render.

No optimistic updates are used. The confirmation flow has a loading state during sequential adds; the list refreshes once at the end. This avoids partial-state visual artifacts.

### Avoiding unnecessary re-fetches

- The AI request is a `useMutation` — it does not re-run in the background or on window focus
- Suggestions are discarded on modal close (no cache to invalidate)
- The `SearchProducts` calls during confirmation use `await` (not React Query), so they do not interact with the query cache

## UX Strategy

### Entry point

Icon button (sparkle/wand) added to the XStack header of `[listId]/index.tsx`, alongside the existing "Comparativo", "Completo", and "Item" buttons. Icon-only with accessible label; small size consistent with sibling buttons.

### Bottom sheet modal

`AISuggestionSheet` is a Tamagui `Sheet` component (existing dependency). It has three internal views controlled by mutation status:

```
Idle / prompt view:
  TextArea for prompt input
  "Sugerir itens" primary button (disabled when input empty)
  Dismiss "×" button

Pending view:
  Spinner + "Gerando sugestões…" message
  Cancel button (dismisses, discards request)

Success view:
  ScrollView of SuggestionItemRow components (checkbox + name + quantity)
  "already_on_list" items shown with muted color + "já na lista" badge, pre-deselected
  "Adicionar [N] itens" primary confirm button (disabled when none selected)
  "Tentar novamente" secondary link (clears results, shows prompt again)

Error view:
  Error message ("Não foi possível gerar sugestões. Tente novamente.")
  "Tentar novamente" button (returns to prompt view)
  Dismiss button

Empty suggestions view:
  "Nenhuma sugestão gerada para este pedido." message
  "Tentar novamente" button
```

### Perceived performance

- The Edge Function call is typically 1–3 seconds. The pending view shows immediately on submit — no frozen UI.
- A character count hint on the prompt input guides the user toward concise prompts.
- The confirmation flow shows a short loading indicator while items are added sequentially; the list screen updates once after.

## Performance Strategy

### Preventing re-renders

- `SuggestionItemRow` is wrapped in `React.memo`; it re-renders only when its `isSelected` or `suggestion` props change.
- `selectedIds` is a `Set<string>` stored in `useState` — spread-copied on toggle to trigger re-render only for the changed row.
- `AISuggestionSheet` is unmounted (not just hidden) when `isAIAssistantOpen` is false — no idle subscription overhead.

### Sequential addition performance

Adding N items sequentially via `AddShoppingListItem` creates N round-trips. For typical AI suggestion counts (5–15 items), this completes in 1–3 seconds total. A single bulk-insert path is not worth implementing at MVP scale.

## Security & Reliability Strategy

### AI response validation

Every response from the Edge Function passes through `aiSuggestionsResponseSchema.parse()` in `SupabaseAIRepository` before any `SuggestedItem` is constructed. A Zod parse failure throws `AIServiceError` which the mutation catches and surfaces as an error state. The user sees "Não foi possível gerar sugestões" — not a crash.

### API key protection

`ANTHROPIC_API_KEY` is a Supabase Edge Function secret — not in the mobile bundle, not in any `.env` file that ships with the app. The Edge Function validates the user JWT before calling the AI provider.

### Graceful degradation

If the Edge Function returns a non-2xx status or the network times out:
- `SupabaseAIRepository.suggestItems()` throws `AIServiceError`
- `useAISuggestMutation` sets status to `"error"`
- `AISuggestionSheet` shows the error view with a retry option
- The active list and all manual flows remain fully functional — no state is corrupted

### Input sanitization

The user's prompt is passed as-is to the Edge Function. The Edge Function validates input length (max 500 chars) and escapes user input before interpolating into the AI prompt template. This prevents prompt injection from affecting the response schema.

### Phase 2 — Edge Function (AI Server Layer)

**Objective**: Deploy a secure, authenticated Edge Function that accepts a natural language prompt and returns structured product suggestions. This is the AI backend foundation — all subsequent phases depend on it being callable.

**Scope**:

| Action | File |
|--------|------|
| NEW | `supabase/functions/suggest-items/index.ts` — Edge Function implementation |

**Edge Function implementation outline**:
```ts
// 1. Extract + validate JWT via supabase.auth.getUser()
// 2. Parse + validate request body (prompt, context)
// 3. Build prompt with list context and existing item names
// 4. Call Anthropic Claude API with JSON output mode
// 5. Parse AI response → validate against JSON schema
// 6. Return { suggestions: [...] } or error response
```

**Prompt template strategy**:
```
You are a grocery shopping assistant for Brazilian supermarkets.
The user is building a shopping list called "{listName}".
Items already on the list: {existingItemNames or "none"}.
Request: "{prompt}"

Return a JSON object with this exact shape:
{ "suggestions": [{ "name": string, "quantity": integer, "unit": string? }] }
Suggest 5–15 realistic grocery items. Do not repeat items already on the list.
```

**Environment**:
- `ANTHROPIC_API_KEY` set via `supabase secrets set ANTHROPIC_API_KEY=...`
- Local testing: `supabase functions serve suggest-items` with a `.env.local` file

**Dependencies**: None — this phase is standalone.

**Risks**:
- Edge Function cold start adds ~200ms to first request → acceptable for MVP
- Anthropic API rate limits → add retry with exponential backoff (max 2 retries)
- AI response schema deviation → Zod validation in the function before returning

**Rollback**: Delete the function deployment. Mobile app is unchanged in this phase.

**Tests**:
- Manual: `curl -X POST` with valid JWT → verify structured response
- Manual: Missing JWT → verify 401 response
- Manual: Empty prompt → verify 400 response
- Manual: Simulate AI timeout → verify 503 response

**Validation criteria**:
- Function deployed and reachable at Supabase project URL
- Authenticated POST returns valid `{ suggestions: [...] }` JSON
- Unauthenticated POST returns `401`
- Invalid body returns `400`
- Suggestions do not repeat items listed in `existingItemNames`

### Phase 4 — Infrastructure Adapter

**Objective**: Implement `SupabaseAIRepository` that calls the Edge Function via `supabase.functions.invoke()`, validates the response, and maps to `SuggestedItem[]`. After this phase, the full AI request chain is functional end-to-end.

**Scope**:

| Action | File |
|--------|------|
| NEW | `apps/mobile/src/infrastructure/repositories/SupabaseAIRepository.ts` |
| MODIFY | `apps/mobile/src/infrastructure/defaultRepositories.ts` — add `ai: SupabaseAIRepository` |
| NEW | `apps/mobile/src/features/ai-assistant/aiAssistant.schemas.ts` — Zod schema |

**`SupabaseAIRepository.suggestItems()` implementation**:
```ts
async suggestItems(prompt, context): Promise<SuggestedItem[]> {
  const { data, error } = await supabase.functions.invoke("suggest-items", {
    body: { prompt, context },
  });
  if (error) throw new AIServiceError(error.message);
  const parsed = aiSuggestionsResponseSchema.parse(data);
  return parsed.suggestions.map((s) => ({
    ...s,
    id: crypto.randomUUID(),
    status: "pending" as const,
  }));
}
```

**Error taxonomy**:
```ts
class AIServiceError extends Error {
  constructor(
    message: string,
    public readonly code: "unavailable" | "invalid_response" | "unauthorized"
  ) { super(message); }
}
```

**Dependencies**: Phase 2 (Edge Function deployed), Phase 3 (port and domain types).

**Risks**:
- `supabase.functions.invoke()` returns `{ data, error }` — need to check both; error is non-null even on HTTP error codes
- Zod parse failures should produce `AIServiceError("invalid_response")`, not crash

**Rollback**: Remove `SupabaseAIRepository` and revert `defaultRepositories.ts`.

**Tests**:
- Integration: Mock `supabase.functions.invoke` to return valid response → verify `SuggestedItem[]` returned correctly
- Integration: Mock returns invalid schema → verify `AIServiceError` thrown
- Integration: Mock returns network error → verify `AIServiceError` thrown
- Manual: Live test against deployed Edge Function with real prompt

**Validation criteria**:
- `pnpm typecheck` passes
- Integration tests pass with mocked Supabase client
- Live test with real authenticated session returns structured suggestions
